# Architecture

## Overview

Waft is a modular system for managing operating system features through a plugin-based architecture. Plugins provide domain entities (devices, data sources, services) to a central daemon (`waft`), which routes entity data and actions between plugins and consumer apps. Apps render UI based on entity types — plugins never describe UI directly.

## Communication Model

All communication flows through `waft` via Unix sockets using length-prefixed JSON (4-byte big-endian length + payload). `waft` acts as the router for all messages between plugins and apps. Message ordering is guaranteed per connection (Unix socket stream semantics).

```
┌──────────┐     Unix socket     ┌──────┐     Unix socket     ┌──────────────┐
│  Plugin   │ ←───────────────→  │ waft │ ←───────────────→  │  App (UI)    │
│ (daemon)  │                    │      │                     │              │
└──────────┘                    └──────┘                     └──────────────┘
                                  ↑  ↑
┌──────────┐     Unix socket     │  │     Unix socket     ┌──────────────┐
│  Plugin   │ ←──────────────────┘  └──────────────────→  │  App (UI)    │
│ (daemon)  │                                              │              │
└──────────┘                                              └──────────────┘
```

### Entity URN Format

Entities are identified by a URN with hierarchical structure:

```
{plugin}/{entity-type}/{id}
{plugin}/{entity-type}/{id}/{child-entity-type}/{child-id}
```

- `plugin` — source plugin name (e.g. `networkmanager`)
- `entity-type` — kebab-cased type name (e.g. `network-adapter`)
- `id` — instance identifier (e.g. `wlan0`)

Nesting depth is not limited but typically one or two levels.

Examples:
- `networkmanager/network-adapter/wlan0`
- `bluez/adapter/hci0/device/AA:BB:CC:DD:EE:FF`

### Subscription Model

Apps subscribe to root entity types (e.g. `adapter`). Subscribing delivers:

1. **Initial state** — `waft` requests current state from the plugin and delivers all existing entities of the subscribed type (and their children) to the app immediately.
2. **Ongoing updates** — when a plugin updates any entity under the subscribed root type, `waft` forwards the update to all subscribed apps.

Updates are typed and URN'd to their actual entity type, not the subscription root. For example, subscribing to `adapter` delivers both adapter updates and device updates as separate messages, each with its own URN and entity type:

- `EntityUpdated { urn: "bluez/adapter/hci0", entity_type: "adapter", ... }`
- `EntityUpdated { urn: "bluez/adapter/hci0/device/AA:BB:CC", entity_type: "device", ... }`

No polling — all updates are push-based.

Apps must wait for the initial state delivery before processing ongoing updates. Events received before the initial state response are ignored — the initial state already reflects the current truth.

### Message Types

#### App → `waft`

- **Subscribe** — app subscribes to a root entity type; triggers initial state delivery
- **Unsubscribe** — app unsubscribes from an entity type
- **Status** — app requests current state for an entity type (same as the implicit request during subscribe, available as a separate message for future flexibility)
- **TriggerAction** — app sends an action on an entity, identified by a UUID; `waft` forwards to the owning plugin and tracks the UUID until acknowledgment. Optionally includes `timeout_ms` to override the default 5-second action timeout

#### `waft` → Plugin

- **CanStop** — asks the plugin whether it is safe to stop (no active side effects)

#### Plugin → `waft`

- **EntityUpdated** — plugin sends updated entity state, `waft` forwards to subscribers
- **EntityRemoved** — plugin signals an entity no longer exists (e.g. USB adapter unplugged, device out of range), `waft` forwards to subscribers; apps drop UI for that entity
- **ActionSuccess** — plugin acknowledges a completed action by UUID
- **ActionError** — plugin reports a failed action by UUID with an error description
- **StopResponse** — responds to `CanStop` with `true` (safe to stop) or `false` (has active work)

#### `waft` → App

- **EntityUpdated** — forwarded from plugin
- **EntityRemoved** — forwarded from plugin
- **ActionSuccess** — forwarded from plugin, matched by UUID
- **ActionError** — forwarded from plugin or generated by `waft` on plugin crash/timeout, matched by UUID
- **EntityStale** — entity data may be outdated (the plugin crashed; `waft` is attempting restart)
- **EntityOutdated** — entity data is definitely outdated (the plugin crashed 5 or more times within the last 60 seconds; `waft` has stopped restarting it)

### Action Tracking

Each `TriggerAction` message carries a UUID. `waft` holds the UUID in a pending set and routes it to the owning plugin. The UUID is resolved when:

- The plugin sends `ActionSuccess` or `ActionError` — `waft` forwards to the originating app and forgets the UUID
- The plugin crashes before responding — `waft` sends `ActionError` to the app with a plugin crash reason
- The action times out — if no acknowledgment arrives within the timeout, `waft` sends `ActionError` to the app and forgets the UUID. The default timeout is 5 seconds. Apps may override the timeout per action by including a `timeout_ms` field in the `TriggerAction` message

## `waft` Daemon

Central daemon that manages the plugin lifecycle and routes messages between plugins and apps.

### Responsibilities

- **Plugin discovery** — locates plugin binaries (`waft-*-daemon`) in a directory determined by `WAFT_DAEMON_DIR` env var, or auto-detected from `./target/{debug,release}`, falling back to `/usr/bin`
- **Plugin registry** — runs `{binary} provides` on each discovered plugin to learn which entity types and schemas it provides; caches the result
- **Schema serving** — provides entity type schemas to apps on request (apps never interact with plugin binaries directly)
- **Plugin spawning** — starts plugin daemons on demand when an app first subscribes to an entity type the plugin provides. On-demand spawning means apps may experience initial latency while a plugin starts and gathers state (e.g. D-Bus enumeration, network scanning). This is acceptable — apps are responsible for showing appropriate loading UI until the initial state arrives
- **Plugin supervision** — monitors plugin health, restarts crashed plugins, sends `EntityStale` and `EntityOutdated` to subscribers on persistent failure
- **Subscription tracking** — maintains which apps are subscribed to which entity types
- **Message routing** — forwards entity updates from plugins to subscribed apps, forwards actions from apps to the owning plugin. Entity updates are debounced per URN — when a plugin sends multiple updates for the same entity in rapid succession, `waft` coalesces them and forwards only the latest state to subscribers
- **Action tracking** — tracks pending action UUIDs, handles timeout and crash scenarios
- **App disconnection** — when an app disconnects (gracefully or not), `waft` removes all its subscriptions and resolves all its pending action UUIDs with `ActionError`. If no subscribers remain for a plugin's entity types, `waft` initiates graceful stop

### Plugin Discovery

At startup, `waft` scans the daemon directory for `waft-*-daemon` binaries and runs `{binary} provides` on each. The output is a JSON document listing entity types. The daemon directory is determined by: `WAFT_DAEMON_DIR` env var, then `./target/debug` or `./target/release` (if they contain plugin binaries), then `/usr/bin`.

Discovery details:
- All binaries are scanned in parallel with a 5-second timeout per binary
- If a binary fails, hangs, or times out, it provides nothing and `waft` logs a warning — other plugins are not affected

### D-Bus Activation

`waft` registers on D-Bus as `org.waft.Daemon`. When an app starts and `waft` is not running, the app requests D-Bus activation to start `waft`. D-Bus guarantees only one instance is activated even if multiple apps request it simultaneously.

After activation, the app connects to the `waft` Unix socket. If the socket is not yet available (waft is still initializing), the app retries with exponential backoff (100ms, 200ms, 400ms, ...) up to a maximum of 10 seconds before reporting an error.

This is the only D-Bus interaction for apps — all subsequent communication uses Unix sockets.

### Design Constraints

- Must be as simple and reliable as possible — it is a single point of failure
- Primarily a router — owns plugin lifecycle, action tracking, and subscription management, but never transforms or aggregates entity data
- If `waft` crashes, all plugin daemons go down with it; apps use D-Bus activation to restart `waft`. This is intentional — plugins are restarted on demand based on active subscriptions, so the system recovers cleanly. Letting orphaned plugins survive a `waft` crash would require reconnection logic and stale state reconciliation with no clear benefit

### Security

Unix socket file permissions provide user-level isolation — sockets live in `/run/user/{uid}/waft/` and are only accessible by the owning user. All plugins and apps run as the same UID. No additional authentication layer is needed for the desktop use case.

## Apps

Apps provide user interfaces. They subscribe to entity types through `waft` and render UI based on the entity data they receive. Each app maps entities to UI independently — different apps present different views of the same entity data (e.g. overview shows a compact toggle, settings shows a full configuration panel). The shared `waft-ui-gtk` library provides reusable GTK widgets, but the decision of which widget to use for which entity is always app-specific.

## Plugins

Plugins run as daemon processes spawned by `waft`. Each plugin manages a specific piece of system software and translates its domain into entity types defined in `waft-protocol`. Plugins have no knowledge of UI — they provide data and handle actions.

### Plugin Binary Interface

Each plugin is a standalone binary that supports the following commands:

- `{binary} provides` — outputs JSON listing entity types and their schemas; used by `waft` for discovery
- `{binary}` (no args) — starts the daemon, connects to `waft` via Unix socket

### Plugin Lifecycle

1. An app subscribes to an entity type (e.g. `network-adapter`)
2. `waft` looks up which plugin provides that entity type (from cached `provides` output)
3. `waft` spawns the plugin daemon if not already running
4. Plugin connects to `waft`, registers its entities
5. `waft` requests current state and forwards entity data to the subscribing app
6. When no apps are subscribed to a plugin's entity types, `waft` initiates a graceful stop (see Plugin Stopping below)

### Plugin Stopping

When no apps are subscribed to a plugin's entity types, `waft` attempts a two-phase graceful stop:

1. `waft` sends a `CanStop` message to the plugin
2. The plugin responds `true` (safe to stop) or `false` (has active work — e.g. inhibiting sleep, maintaining a VPN connection)
3. If `true`, `waft` stops the plugin. If `false`, `waft` retries after 30 seconds

This is the only polling mechanism in the system. `waft` repeats the `CanStop` check every 30 seconds until the plugin consents or an app re-subscribes.

### Crash Recovery

- `waft` detects plugin crash and immediately sends `EntityStale` to all subscribers of that plugin's entity types
- `waft` restarts the plugin automatically; on success the plugin re-registers its entities and apps receive fresh `EntityUpdated` messages
- If the plugin crashes 5 or more times within a 60-second window, `waft` stops restarting it and sends `EntityOutdated` to subscribers
- Pending action UUIDs for the crashed plugin are resolved with `ActionError`
- Apps may choose to show degraded UI for stale entities or hide UI entirely for outdated entities

### Compatibility

- **Forward compatibility** — when a plugin registers an unknown entity type, apps that don't understand it simply ignore it. The entity sits unused and harmless. When a plugin adds a field to an entity type, apps with an older protocol version ignore the unknown field.
- **Backward compatibility** — when an app subscribes to an entity type that no plugin provides, `waft` has nothing to start. The app does not render UI for that entity type.
- **Protocol versioning** — the protocol version is the `waft-protocol` crate version. Entity type schemas evolve additively (new optional fields). Removing or renaming fields is a breaking change handled by a major version bump of `waft-protocol`. No runtime version negotiation — all components are expected to use the same protocol version.

## `waft-protocol`

Shared package defining the contract between plugins, `waft`, and apps.

### Contains

- Entity type definitions (Rust types with serde serialization)
- URN format and parsing (hierarchical)
- Message types (subscribe, unsubscribe, status, trigger-action, entity-updated, entity-removed, action-success, action-error, entity-stale, entity-outdated, can-stop, stop-response)
- Wire format specification (length-prefixed JSON)
- Action UUID generation

### Does NOT Contain

- UI rendering logic (app-side concern)
- Plugin implementation details (plugin-side concern)

### Entity Type Design

Entity types that have variants use a tagged enum (`kind` discriminator) rather than splitting into separate entity types. This allows apps to subscribe to one type and handle variants.

```rust
struct NetworkAdapter {
    id: String,
    name: String,
    active: bool,
    ip: Option<IpInfo>,
    kind: AdapterKind,
}

enum AdapterKind {
    Wired {
        profiles: Vec<Profile>,
        current_profile: Option<String>,
    },
    Wireless {
        networks: Vec<Network>,
        known_networks: Vec<Network>,
        connected: Option<Network>,
    },
}
```

Serialized as a serde tagged enum in JSON. Unknown variants are ignored by apps (forward compatibility).

## Plugin Example: Network Manager

The `networkmanager` plugin manages network devices and VPN connections using NetworkManager.

### Provided Entities

#### `network-adapter`

Generic adapter info:
- Network (IP, mask, gateway, DNS, public IP)

Generic adapter methods:
- Activate
- Deactivate

Wired adapter info (`kind: Wired`):
- Current profile
- List of profiles

Wired adapter methods:
- Select profile

Wireless adapter info (`kind: Wireless`):
- Available networks
- Available known networks
- Connected network

Wireless adapter methods:
- Scan
- Connect to network

#### `vpn`

VPN methods:
- Disconnect all VPNs
- Connect to specific VPN
- Disconnect from specific VPN
